# Изменения: Автоматическое подтверждение доставки

## Дата: 2026-03-11

## Цель
Убрать необходимость для продавца вручную подтверждать получение заказа. Вместо этого, delivery-service автоматически подтверждает получение заказа покупателем и уведомляет posts-service.

## Изменения в Backend

### 1. posts/order_router.py

#### `/delivery-received` webhook (обновлен)
- **Было**: Обновлял статус заказа на `DELIVERED` и отправлял SMS с просьбой оставить отзыв
- **Стало**: Автоматически:
  - Обновляет статус на `COMPLETED` (сразу завершает заказ)
  - Устанавливает `confirmed_by_buyer = True`
  - Обновляет статистику продавца (`sells_count`, `rating`)
  - Скрывает чаты связанные с заказом
  - Отправляет SMS покупателю с благодарностью и ссылкой на отзыв

#### `/confirm` endpoint (удален)
- **Было**: Покупатель подтверждал получение товара и оставлял отзыв
- **Стало**: Удален полностью

#### `/review` endpoint (новый)
- Позволяет покупателю оставить отзыв на уже завершенный заказ
- Не меняет статус заказа
- Обновляет рейтинг продавца
- Принимает параметры:
  ```json
  {
    "order_id": 123,
    "rating": 5,
    "review_text": "Отличный продавец!"
  }
  ```

### 2. posts/models.py

#### OrderConfirmation (удален)
- Класс больше не используется
- Удален из кодовой базы

#### Order model (без изменений)
- Поля `confirmed_by_buyer` и `rejected_by_buyer` оставлены для совместимости с БД
- Не требуется миграция базы данных

### 3. delivery/delivery_service.py (без изменений)

Существующая реализация уже корректно работает:
- При переводе доставки в статус `picked_up` вызывается `_notify_posts_service_delivery_received()`
- Этот метод отправляет webhook на `/delivery-received` в posts-service
- Отправляется SMS с кодом получения
- Отправляется SMS о получении посылки

## Workflow после изменений

### Текущий процесс заказа:

1. **Покупатель создает заказ** → статус `pending_payment`
2. **Покупатель оплачивает** → статус `paid`
   - Создается доставка в delivery-service
3. **Продавец отмечает "Отправлено"** → статус `shipped`
   - Delivery-service обновляется на статус `in_transit`
4. **Delivery-service симулирует доставку в пункт выдачи**
   - Статус доставки: `at_pickup_point`
   - Отправляется SMS с кодом получения
5. **Покупатель получает товар из пакомата**
   - Статус доставки: `picked_up`
   - **Автоматически**: delivery-service вызывает webhook в posts-service
6. **Posts-service получает webhook** → статус `completed`
   - Устанавливает `confirmed_by_buyer = True`
   - +1 к продажам продавца
   - Обновляет рейтинг (если есть отзыв)
   - Скрывает чаты
   - Отправляет SMS покупателю с ссылкой на отзыв
7. **Покупатель может оставить отзыв** (опционально)
   - Через endpoint `/review`
   - Обновляется рейтинг продавца

## Преимущества

1. ✅ **Автоматизация**: Продавцу больше не нужно подтверждать получение
2. ✅ **Меньше действий**: Покупателю не нужно подтверждать получение вручную
3. ✅ **Чистый код**: Удален неиспользуемый функционал
4. ✅ **Надежность**: Доставка подтверждается автоматически при получении из пакомата
5. ✅ **Отзывы**: Покупатель все еще может оставить отзыв через отдельный endpoint

## Примечания

### Статус "shipped" теперь краткосрочный
После изменений заказ в статусе `shipped` находится недолго:
- Продавец отмечает "Отправлено" → `shipped`
- Delivery-service симулирует доставку (настраивается в configs, по умолчанию несколько часов)
- Покупатель получает из пакомата → автоматически `completed`

Текст "Awaiting confirmation" в my-sales.html технически больше не точен, но оставлен для информации. Можно заменить на более нейтральный текст типа "В доставке" если требуется.
